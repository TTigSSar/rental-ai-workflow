## ADR-001: Chat conversations are scoped to a Booking
Date: 2026-07-07 | Status: accepted
**Decision:** A `Conversation` is 1:1 (owner ↔ renter) and keyed to exactly one `Booking` (`BookingId` unique). The conversation's status pill is **derived** from the linked booking's status + a closed flag (requested→approved→active→return_due→closed), not stored. Per-participant read state (unread count, "Seen") lives in a `ConversationParticipant` row (`LastReadMessageId`/`LastReadAt`), one per user. Inbox-preview fields (`LastMessageAt`, snippet, toy title/image) are denormalised on `Conversation` for join-free list reads — same pattern as `Notification`.
**Why:** The approved DoRent design pins every thread to one toy+booking; system messages are emitted by booking lifecycle events; the read-only lock is "booking complete AND both reviews in". Booking is the natural key. Denormalising the preview + a participant read-cursor matches the codebase's existing join-free-read style and directly powers the unread badge and Seen receipt.
**Rejected:** (a) Conversation keyed by user-pair — breaks the design's per-booking system messages and status pill, and the existing Angular model already carries `listingTitle`. (b) Storing read state as two nullable columns on `Conversation` (OwnerLastReadAt/RenterLastReadAt) — simpler for strict 1:1 but doesn't generalise and reads awkwardly for the "Seen on my last message" query; the participant table is cleaner.
**Frontend impact:** the existing minimal Angular chat model (`title`/`participantName`/`listingTitle`/text-only) is replaced to match the design (booking-scoped, status, image/system messages). Contract rewrite handled by contract-guardian in a later Phase-1 increment.
**Amended 2026-07-11:** the pill vocabulary gained a `completed` state: requested → approved → active → return_due → **completed** → closed. `completed` = booking Completed but `ClosedAt` still null (chat open, awaiting both party reviews); `closed` is emitted ONLY from the `ClosedAt` override. Rationale: M-007's fix mapped Completed straight to `closed`, which made the pill read "Closed" on a conversation that still accepted messages — the vocabulary must distinguish "rental done" from "thread locked".

## ADR-002: Chat image attachments are served as unauthenticated capability URLs
Date: 2026-07-11 | Status: accepted (**deliberate trade-off, not an oversight**)
**Decision:** Chat attachments are written to a conversation-scoped directory with a server-generated filename and served by the same `UseStaticFiles` pipeline as listing images — i.e. **anyone holding the URL can fetch the file without being a participant, or even authenticated**. The URL is unguessable (GUID conversation path + GUID filename), so possession of the URL *is* the authorization ("capability URL").
**Why:** the human weighed it explicitly against the alternative below and chose to ship the vertical. It keeps the read path a plain `<img [src]>`, browser-cacheable, and identical in shape to listing images.
**Known consequence (accept it knowingly):** chat attachments are **private content between two users**, unlike listing images which are public by design. A URL that leaks — browser history, `Referer`, server logs, a forwarded screenshot — grants permanent, unrevocable access to that image. There is no participant check. `/security-review` will flag this; that finding is expected, and this ADR is the answer to it.
**Rejected:** an authenticated streaming endpoint (`GET api/chat/conversations/{id}/attachments/{file}`) that verifies the caller is a participant. Correct on privacy, but `<img src>` does not send the `Authorization` header, so the client would have to fetch each image as a blob and hand the component an object URL — more moving parts, no browser caching, and roughly half a day of extra work. Deferred, not dismissed: revisit before any real (non-demo) launch.
**Related infra:** `docker-compose.yml` mounts a `chat-uploads` volume at `/app/wwwroot/uploads/chat`; without it attachments vanish on image rebuild.

## ADR-003: Staging stand = single VPS + docker compose + Cloudflare Tunnel, one same-origin hostname
Date: 2026-07-13 | Status: accepted for the **infrastructure shape**, but the "staging stand" itself was **never deployed and never will be — see ADR-004**.
> ⚠️ `test.dorent.am` **does not exist** (NXDOMAIN; the tunnel has no route for it) and `docker-compose.staging.yml` has never been run. The VPS runs **production only**, on the apex — see the 2026-07-13 amendment at the bottom of this ADR. Everything about the *shape* (single VPS, docker compose, token Cloudflare Tunnel, same-origin nginx reverse proxy, zero inbound ports) held and went to production unchanged; only the word "staging" is fiction.
**Decision:** The test stand for `dorent.am` runs the whole stack (SQL Server, API, UI, cloudflared) via `rental-api/docker-compose.staging.yml` on one cheap VPS (Hetzner CPX12; the actual IP is deliberately not recorded here — this repo is public). Exposure is a token-based **Cloudflare Tunnel** (outbound-only; zero inbound ports, TLS by Cloudflare). One public hostname `test.dorent.am` → the UI's nginx, which already reverse-proxies `/api/`, `/uploads/`, `/hubs/` to the API container — same-origin, so **no CORS config and no API subdomain are required** (`api.dorent.am` is optional, Swagger convenience only). `ASPNETCORE_ENVIRONMENT` stays `Development` on the stand deliberately (dev seed + Swagger); secrets move to `.env`. Guide: `rental-api/DEPLOY-STAGING.md`.
**Why:** SQL Server (~2 GB RAM) rules out free tiers and Cloudflare Workers; the dev compose already models the full stack; the domain was already on Cloudflare, making the tunnel the shortest path to HTTPS with no nginx/certbot/ports on the host.
**Rejected:** (a) UI on Cloudflare Pages + API on VPS — splits origins, forcing CORS + a rebuilt `apiBaseUrl`, for no benefit at staging scale. (b) Azure App Service + Azure SQL — costlier and slower to set up for a throwaway stand. (c) Classic A-record + nginx/certbot on the host — works, but needs open 80/443 and cert management the tunnel gives for free.
**Amended 2026-07-13:** the same architecture went to **production**: `docker-compose.production.yml` (ASPNETCORE_ENVIRONMENT=Production — no seed/Swagger; admin via idempotent `Bootstrap__Admin*` bootstrap; api/ui healthchecks; log rotation), public hostnames `dorent.am` + `www` → ui:80 through tunnel `dorent-production`. The 2 GB VPS stays (no budget): 4 GB swap + `MSSQL_MEMORY_LIMIT_MB=1024` — acknowledged as temporary; rescale to 4 GB is the first upgrade when money allows. Daily backups: `deploy/backup-production.sh` (cron 03:30, 7-day retention, `--verify` restore test proven live). Off-site copy still pending. Classic A-record variant (c) was re-proposed by an external checklist and re-rejected — the tunnel already covers Full-strict-equivalent origin encryption with zero open ports.

## ADR-005: The production showcase owner uses the published `Demo1234` password
Date: 2026-07-16 | Status: accepted (**deliberate trade-off, chosen by the human after an explicit warning — not an oversight**)
**Decision:** production's demo catalogue is owned by `showcase@dorent.am`, a real, live, non-admin account on the public site, whose password is **`Demo1234`** — the same password documented in `CLAUDE.md` under "Demo accounts", in a **public** GitHub repo (`TTigSSar/rental-api`). Enabled via `Bootstrap__DemoContentEnabled=true` + `Bootstrap__DemoOwner*` in the server `.env`; the bootstrap creates the owner + the Approved seed listings/images and is idempotent/additive.
**Why:** Tigran wanted a memorable password he does not have to store or look up, weighed against the alternative below, and accepted the risk knowingly. The account is deliberately **not** an admin.
**Known consequence (accept it knowingly):** anyone can read the password in the public repo, and owner emails are visible around listings/conversations — so this account should be assumed compromisable by anyone who cares to try. Whoever holds it can edit or delete the showcase listings, approve/reject bookings made on them, and chat with real renters **as the platform's storefront**. There is no rate limiting or lockout on login, and no password reset path on production (`IEmailService` is a development stub). `/security-review` will flag this; this ADR is the answer to that finding.
**Rejected:** a generated strong password written straight into the server `.env` (never through chat) and copied once into a password manager — objectively safer at near-zero cost, recommended by the orchestrator, declined by the human. Revisit before any real (non-demo) launch, together with ADR-002.
**Related:** demo content is otherwise clean — no other seeded accounts, no bookings/reviews/chats on production. Categories are NOT part of this trade-off: they ship as reference data via migration `20260716103601_SeedReferenceCategories` and exist regardless of the flag.

## ADR-004: There is no remote staging stand — local docker compose IS the test environment
Date: 2026-07-14 | Status: accepted
**Decision:** We do **not** run a remote staging/demo stand. The seeded test environment is the **local** `rental-api/docker-compose.yml` (ASPNETCORE_ENVIRONMENT=Development → idempotent dev seed + Swagger), which already models the full stack. `docker-compose.staging.yml` and `DEPLOY-STAGING.md` describe a stand that was never deployed; they are kept only as a starting point if a stand is ever funded, and are marked as not-deployed.
**Why:** the VPS is a 2 GB Hetzner CPX12 already running production, where SQL Server alone is capped at 1024 MB and a swap file is required merely to build images. A staging stand is a **second full stack including a second SQL Server (~1 GB)** — it does not fit. Deploying it on that box would very likely have the OOM killer take down the *production* database. It would also collide on ports (`127.0.0.1:8080` / `:4200` are already bound by production). The only safe variants cost money (a second VPS, or a rescale to 4 GB), and the stand is currently only needed by one developer who already has the identical stack running locally.
**Never do this:** do NOT "fix" an empty production database by enabling the dev seed there. Production is empty **by design** — `Program.cs` gates `SeedDevelopmentDataAsync()` behind `IsDevelopment()` and gives every other environment the idempotent `BootstrapAdminAsync` instead. The dev seed's accounts (`admin@rental.local` / `owner@rental.local`, password `Demo1234`) are **published in CLAUDE.md** and include an **admin**; seeding them into a live public domain hands over the admin panel to anyone who has read the repo.
**Rejected:** (a) second VPS (~€5/mo) — correct and fully isolated, revisit when a stand is actually needed by someone other than the developer; (b) rescale the current box to 4 GB (~€8/mo) — cheaper, but staging and production would share hardware, so a staging spike can still hurt production; (c) one SQL Server serving both a prod and a staging database — saves the ~1 GB but couples the production database to staging's load and migrations, which is worse than the problem it solves.
**How this bit us (2026-07-14):** ADR-003 read as though the stand existed, so an empty production database was diagnosed as "the seed didn't run" and the fix was nearly "add the missing DNS record for `test.dorent.am`". It resolves to NXDOMAIN because nobody ever deployed it. This is the M-001 class — a document describing intent was read as describing reality.

## ADR-006: Specialist subagent models — Sonnet by default, Opus only for platform-engineer
Date: 2026-07-19 | Status: accepted
**Decision:** Every specialist agent in `.claude/agents/` declares its model explicitly in frontmatter. `backend-dev`, `frontend-dev`, `contract-guardian`, `verifier`, `qa-engineer` → `sonnet`. `platform-engineer` → `opus`. No agent is left without a `model:` key. Opus for an implementation agent is now an explicit per-spawn override by the orchestrator (architectural work — store shape affecting several features, routing refactors), not a standing default.
**Why:** Claude Code's usage panel (2026-07-19) attributed **67% of a day's consumption to `frontend-dev` alone**, against 1% for `backend-dev` — the two agents do work of the same class, and the only difference was `model: opus` vs `model: sonnet`. `qa-engineer` and `platform-engineer` had **no `model:` key at all**, silently inheriting the orchestrator's Opus; harmless while their usage was ~2%, but a trap for the pending backups/monitoring work on dorent.am. `platform-engineer` keeps Opus deliberately: it is the only agent acting on live production (SSH, compose, backups, Cloudflare), where the cost of a wrong action is an outage, not a retry.
**Known consequence:** the claim "Sonnet is sufficient for Rental-Ui work" is a **hypothesis, not yet validated on this codebase**. If UI tasks start costing extra iterations or drifting from `Rental-Ui/CLAUDE.md` conventions, revert `frontend-dev` to Opus — and then look for the saving in session shape instead (see below), not in model choice.
**Rejected:** (a) leave everything on Opus — the workflow in `CLAUDE.md` §2 mandates subagents for all code, so ~100% of consumption flows through them; the model choice there *is* the cost model. (b) all-Sonnet including `platform-engineer` — saves little (it was ~2–3%) and spends the saving in exactly the place where a mistake is most expensive. (c) attack session length/context instead of models — real but secondary and behavioural, not configurable: the same panel showed 57% of weekly usage at >150k context and 56% from 8h+ sessions, addressed by `/clear` between unrelated tasks rather than by any file in this repo.
**Note on the figures:** the usage panel states they are approximate, cover local sessions on this machine only, and are independent characteristics rather than a partition — treat them as direction, not accounting.

## ADR-007: Maps run on Leaflet + raster OpenStreetMap tiles, behind one `app-map` component
Date: 2026-07-21 | Status: accepted (**for Phase 1; the trilingual-label requirement is knowingly unmet — see below**)
**Decision:** The map stack is **Leaflet 1.9 with raster tiles from `https://tile.openstreetmap.org/{z}/{x}/{y}.png`**, no account, no API key, no style JSON. Leaflet is imported in exactly one file — `Rental-Ui/src/app/shared/ui/map/map.component.ts` (`app-map`) — via a dynamic `import('leaflet')`, with `leaflet.css` injected lazily as a `<link>` to `/leaflet/leaflet.css` (copied at build time by an `angular.json` assets entry, not bundled). Nothing else in the codebase may import Leaflet, and no Leaflet type crosses the component boundary: the public API is `center` / `zoom` / `pin` / `height` / `interactive` / `crosshair` / `circleRadiusMeters` in, `centerChange` / `mapError` out. The OSM attribution control is always visible — that is the ODbL license condition, not decoration.
**Why:** the approved design (`listing-location.jsx` in the claude.ai design project) is itself built on Leaflet + OSM raster, so this is 1:1 with what was signed off. Measured cost: **+4.27 kB** on the main bundle, with Leaflet itself in a separate lazy chunk (149.55 kB raw / 37.58 kB gzip) that loads only when a map actually mounts. Zero vendor accounts to create, zero keys to proxy, zero spend — and the exit is a one-URL swap because raster XYZ endpoints are interchangeable.
**Known consequence (accept it knowingly):** (a) **Trello card P0-3's trilingual-label requirement is not met.** OSM's standard raster style labels Yerevan in Armenian and Latin only — there is no Russian layer and no `coalesce(name:hy, name:ru, name)` expression, because raster tiles carry no queryable labels at all. RU-locale users see Armenian/Latin map labels. This is the price of skipping vector tiles and was accepted deliberately. (b) `tile.openstreetmap.org` is a **volunteer-funded service whose tile usage policy excludes heavy or commercial traffic**. At DoRent's current volume this is within fair use, and the detail page is tap-to-load specifically so a page view does not fetch tiles — but this stops being acceptable well before the platform is busy. Both are resolved by the same future move: MapLibre GL + a paid vector provider (MapTiler/Geoapify) with a versioned style JSON in the repo, which is what P0-3 originally specified.
**Rejected:** MapLibre GL + vector tiles now — it is the technically correct answer and the only one that delivers trilingual labels, but it needs a provider account, a key proxied through our API, a style JSON to own and maintain, and ~200 kB of library, for a Phase-1 slice whose job was to get two designed screens working. Deferred, not dismissed; revisit when either the label gap or the tile-policy limit actually bites.

## ADR-008: A listing's exact location is stored but never published — geohash-6 fuzzing is the public coordinate
Date: 2026-07-21 | Status: accepted
**Decision:** Three distinct location facts, three audiences.
- `Listings.Latitude`/`Longitude` — the exact pin the owner dropped. Visible to the **owner and admins only**.
- `Listings.PublicLatitude`/`PublicLongitude` — the **centroid of the geohash-6 cell** containing that pin, computed at write time. This is what every other caller receives, in the same `latitude`/`longitude` DTO fields. Measured cell size at Yerevan's latitude: **933 m east-west × 611 m north-south** (longitude cells narrow by cos(latitude); the textbook 1.22 km figure is equatorial).
- `Listings.DistrictId` — point-in-polygon of the exact pin against a committed OSM boundary asset. Public; district granularity is not sensitive.
`AddressLine` follows the **contact-reveal** gate instead: owner, admin, or a renter whose booking on that listing reached `Approved`/`Active`/`Completed` — the renter who is actually coming to collect the toy.
**Why:** these are family homes with children's toys. A raw coordinate is a reverse-geocodable home address, so the only safe design is one where the exact value never leaves the server for an unauthorized caller — not rounding at render time, not obscuring in the UI. Snapping at **write** time (rather than fuzzing per request) is what makes it hold up: every observer of a given listing sees the identical published point forever, so repeated or multi-account observation yields no additional bits and cannot be averaged back toward the truth. Two different exact points inside one cell publish identically. There is a test for exactly this.
**Consequences to respect:** (a) the detail map draws a **600 m** circle — deliberately not the design prototype's decorative 520 m — because a circle smaller than the real uncertainty claims precision we do not have, which is the failure this whole feature exists to prevent. (b) Any future distance filter, distance sort or map-pin endpoint (Phase 2, cards P2-1/P2-3) **must** compute from the public pair and round its output; computing a "rounded" distance from the exact pin re-leaks it through arithmetic. There are comments at the fuzzing seam saying so. (c) `latitude` in `ListingDetailsResponse` no longer means "where the toy is" — it means "where the toy is, to the precision you are entitled to". The frontend model documents this at the field; it is the most misuse-prone thing in the feature.
**Rejected:** (a) returning `null` coordinates to the public and showing no map — that was the H1 hotfix's interim state, and it makes the listing page strictly worse for renters who legitimately want to know the neighbourhood. (b) Per-request random jitter around the true point — trivially defeated by averaging repeated reads. (c) Publishing the district only, with no coordinates — safe, but loses the "roughly here in the district" information that makes the map worth showing at all.
**How this nearly bit us (2026-07-21):** the map, the fuzzing and the owner-facing promise ("only the approximate area is published, never your exact address", shipped in three languages) all went in while `AddressLine` was still being returned **ungated** to anonymous callers — the exact street address sat in the same JSON response as the deliberately-fuzzed coordinates. Card H1's own description asserted that field was already gated; it was not, and nobody checked, because the assertion sounded authoritative. Found only by an agent auditing adjacent code during the last card of the phase, and fixed in `875790f`. M-001 class: a document describing intent was read as describing reality.
