## M-001: rental-api/README.md is outdated — do not treat it as truth
Date: 2026-07-06 | Area: backend
**Symptom:** README describes an early toys-MVP: no reviews, no notifications, no booking activate/complete/cancel, no listing Draft/Archived statuses, Angular version wrong.
**Cause:** README was written for the MVP milestone and never updated as features landed.
**Rule:** source of truth = controllers + `api-contract.ts` + domain enums. Update or delete stale README sections when touching a described area.

## M-002: A delegated agent seeded the EXEMPLAR feature, not the requested one
Date: 2026-07-07 | Area: infra (agent workflow)
**Symptom:** backend-dev was asked to add a **chat conversation** dev seed. It instead wrote a `SeedNotificationsAsync` method seeding NOTIFICATIONS (the feature it had read as a pattern exemplar), wired it into the seed runner, then died on a session limit before adding the requested get-or-create route. Orchestrator caught it on review and reverted the off-task change.
**Cause:** the spec pointed the agent at the notifications vertical as a "mirror this pattern" exemplar; the agent conflated "mirror the pattern" with "seed that feature's data." Cold subagents blur exemplar-vs-target when both are named in the same task.
**Rule:** when a spec cites an exemplar feature, state explicitly "reproduce the *shape*, do NOT create the exemplar's data/entities." Always verify a delegated agent's diff against the spec before integrating — never commit an agent's output unreviewed. Keep seed/data tasks separate from pattern-study tasks.

## M-003: Documented dev-seed password was wrong (`LocalDemo123!` ≠ `Demo1234`)
Date: 2026-07-07 | Area: infra (docs vs code)
**Symptom:** every seed-account login returned 401 during live verification. Docs (rental-api README, root CLAUDE.md, verifier.md, the session's demo-accounts note) all said the password is `LocalDemo123!`; the actual value in `DevelopmentSeedCredentials.cs` (const `Password`) is **`Demo1234`**.
**Cause:** the seed password was changed in code at some point; the docs were never updated, and CLAUDE.md/agent files inherited the stale value.
**Rule:** source of truth for the seed password is `DevelopmentSeedCredentials.Password`, not any doc. Fixed CLAUDE.md + verifier.md to `Demo1234`. When an agent (esp. verifier) needs seed creds, it should read that constant rather than trust docs.

## M-004: `takeUntilDestroyed()` bare in `ngOnInit()` → NG0203 runtime crash (green build/tests)
Date: 2026-07-07 | Area: frontend
**Symptom:** the chat thread page rendered blank on all viewports. `dotnet`/`npm` builds AND all unit tests were green — the failure only surfaced in the live walk (screenshots). Root cause: `takeUntilDestroyed()` called bare inside `ngOnInit` throws `NG0203: can only be used within an injection context`.
**Cause:** `takeUntilDestroyed()` needs an injection context (constructor / field initializer) unless a `DestroyRef` is passed explicitly. `ngOnInit` is not an injection context. Compiles fine; crashes at runtime. The chat feature had **zero unit tests**, so nothing caught it before the live walk.
**Rule:** call `takeUntilDestroyed()` in a field initializer/constructor, or pass `takeUntilDestroyed(this.destroyRef)` with an injected `DestroyRef`, when used in a lifecycle hook. Every new feature needs at least one component-render spec (`fixture.detectChanges()`) — it catches injection-context and template crashes that build+typecheck miss. This is why the two-tier verification's **live walk with screenshots** is mandatory: build+tests were all green here.

## M-005: Docker API crashes on restart — EF MigrateAsync race vs just-started SQL Server
Date: 2026-07-07 | Area: infra (docker)
**Symptom:** `docker compose up --build -d` after a `down` (volume kept) → `rental-api-api-1` exits **139**; API log shows `CREATE DATABASE [RentalPlatformDb]` → `SqlException (1801): Database 'RentalPlatformDb' already exists`, thrown inside `MigrateAsync` at `MigrationExtensions.ApplyMigrationsAsync` (unhandled → process dies). db + ui come up fine; api is missing from `docker compose ps`.
**Cause:** the db container's healthcheck is only `SELECT 1` on master, which passes before the persisted user DB is fully online. EF Core `Database.MigrateAsync()` runs its existence check in that window, gets "does not exist", tries `CREATE DATABASE`, and the DB *is* there → 1801 → unhandled crash. It's a startup race, NOT a code/schema bug — and **unrelated to whatever app change triggered the rebuild** (here it was a frontend-only nav icon).
**Rule:** operational fix — once db is fully online, restart just the API: `docker compose -f rental-api/docker-compose.yml up -d api` (succeeds on the retry). `down -v` (fresh volume) also avoids it but wipes data. **Proper fix (follow-up):** harden `MigrationExtensions.ApplyMigrationsAsync` with a retry loop / `EnableRetryOnFailure` so migrate retries transient SQL-startup errors instead of crashing the process. Don't chase it in app code — check the API log for 1801 first.

## M-006: Realtime echo + optimistic send → own message rendered twice (dedupe must be symmetric)
Date: 2026-07-08 | Area: frontend (realtime)
**Symptom:** with SignalR realtime chat, a user's OWN just-sent message rendered as two identical bubbles (persisted until reload); the server had only one record. All three live realtime scenarios otherwise PASSED — the two-session verifier walk caught this; the unit tests initially didn't (they tested the single, non-racing append path).
**Cause:** two append paths for the open thread — `sendMessageSuccess` (HTTP POST response) and `realtimeMessageResolved` (SignalR `messageReceived` echo, which the server broadcasts to BOTH participants incl. the sender). Only the realtime handler deduped by `id`; `sendMessageSuccess` appended unconditionally. When the hub broadcast beat the POST response (a real, fairly consistent race), the message was appended twice.
**Rule:** when a message can arrive via more than one channel (optimistic local echo, HTTP response, realtime push), **every** append path must dedupe by a stable server id — not just one. Broadcasting the sender's own message back over the socket is fine (simpler server), but the client must treat its own send and the echo as the same message. Add a regression test that seeds the message as already-present and asserts the second path is a no-op.
*(Note: an earlier M-006 on UTC DateTime serialization was removed from this file during a manual edit; that fix still lives in rental-api commit `3d21636`. Ask if you want it restored here.)*

## M-007: Chat header status pill stuck on "Active" after completion — TWO root causes, and the orchestrator's first hypothesis was wrong
Date: 2026-07-08 | Area: backend + frontend (FIXED — rental-api `b2269d0`, Rental-Ui `5fca9cf`)
**Symptom:** the booking-context header pill atop the chat thread kept reading "Active" after the booking was Completed, and survived a full page reload. Surfaced by the verifier as a side-observation during the chat-system-messages walk.
**Cause (two, independent):** (1) **Backend, the real root cause** — `ChatTokens.StatusToken` mapped `BookingStatus.Completed => "active"`. The `closedAt`-override that would return `"closed"` never fires because the `Conversation.ClosedAt` auto-lock was never built. So the *server itself* derived `"active"` for a completed booking — the pill was stale even on reload. (2) **Frontend** — the pill binds `conversation.status`, set only by `loadConversationDetails`; realtime system messages appended to `messages` but never refreshed `status`, so even a corrected value wouldn't appear until reload.
**Rule:** the fix needed BOTH: backend `Completed => "closed"` (contract-safe — client already renders the token) for the value, and a frontend re-fetch on system-message arrival (`realtimeMessageReceived$` → `loadConversationDetails` when `isOpen && type==='system'`) for the live update. **Process lesson (the important one):** the orchestrator delegated the frontend fix on the hypothesis "re-fetch will re-derive the correct status" — that hypothesis was FALSE; the server returned the wrong value, so re-fetch alone rendered "Active" faithfully. The delegated agent's **live verification grounded against `GET /api/chat/conversations/{id}` (server truth)** falsified the orchestrator's root-cause assumption and found the real backend bug. Lesson: when a "derived" value looks stale, verify the derivation SOURCE (hit the API, read the token the server actually returns) before assuming the staleness is a client refresh problem. Don't trust the orchestrator's root-cause guess — make the verifier prove it against ground truth.

## M-008: SignalR realtime is dead in the docker stack — UI nginx doesn't proxy/upgrade the hub (405 on negotiate)
Date: 2026-07-08 | Area: infra (docker) — FIXED 2026-07-11 (Rental-Ui: `location /hubs/` + `nginx-map.conf` http-level `map` include; verified live in docker: negotiate 200, WS frames delivered to both parties)
**Symptom:** in the docker deployment the browser fails the SignalR hub negotiate with `Failed to complete negotiation … Status code '405' (Not Allowed)`; `messageReceived` never arrives, so realtime chat (new messages, system lines, the M-007 live pill flip) silently does nothing. Everything works under `npm start` (Angular dev proxy) — which is why every prior "realtime live" verification (realtime feature, system messages, M-007) passed: they all ran in dev, never against the docker UI container.
**Cause:** the UI container's nginx serves the SPA but has no `location` for the hub path (`/hubs/chat`) with the WebSocket upgrade headers (`Upgrade`/`Connection`, `proxy_http_version 1.1`) proxied to the API — so the hub endpoint returns 405 and the WS never upgrades.
**Rule:** OPEN backlog item — add an nginx `location /hubs/` block in the UI image proxying to the API service with WS upgrade headers (and confirm the API CORS/hub auth via `?access_token=` still works through it). Until then, realtime only works in dev. **Meta-lesson:** "verified live" is only as good as the environment — our live walks ran in dev, so a docker-only infra gap went unseen for three features. When a feature has a prod/docker deployment, verify at least once IN that deployment, not only in dev.

## M-009: Two-pane shell crashed on COLD direct-nav to /chat/:id — and skipping the independent verifier let it ship
Date: 2026-07-09 | Area: frontend + process
**Symptom:** direct-loading a thread URL `/chat/{conversationId}` in a fresh tab (no prior `/chat` visit) threw `Cannot read properties of undefined (reading 'paramMap')` and left the thread pane blank (footer showing). Warm nav (inbox → row click) worked, so it was invisible in normal use.
**Cause:** `ChatShellComponent.threadOpen` (the signal driving mobile single-column show/hide + the desktop empty pane) was seeded in a field-initializer by walking `this.route.firstChild` and reading `child.snapshot.paramMap`. On cold direct-nav a child `ActivatedRoute.snapshot` isn't populated yet during construction → throws. Fix: derive thread-open from the **router URL** instead (`isThreadOpenUrl(url)` = non-empty segment after `/chat/`), used identically in `startWith`/`initialValue`/`NavigationEnd` — no snapshot race, mirrors the app-level `isChatUrl`.
**Rule:** never read a child route's `snapshot` in a parent's field-initializer/constructor — it isn't ready on cold load; derive route-shape state from `router.url` (a string) instead. **Process lesson (the real one):** this shipped in the two-pane shell increment where the orchestrator MERGED ON THE IMPLEMENTER'S OWN LIVE WALK + a diff review, skipping an independent `verifier` pass to save a cycle — and the implementer only ever tested warm nav, so the cold-nav crash slipped. The independent verifier exists precisely to test the paths the implementer didn't think of (here: cold direct-nav). For any ROUTING/architectural change, run the independent verifier before merge even if the implementer self-verified — the classes of bug it catches (M-004, M-006, this) are exactly the ones the author is blind to. Also: every route-shape branch needs a cold-load test (seed the URL, assert no throw), which is why this had no regression test.

## M-010: A "derived read-flag" with no write-path — chat `IsClosed`/`ClosedAt` never fired (2nd instance of the M-007 pattern)
Date: 2026-07-09 | Area: backend (FIXED — rental-api `16237bc`)
**Symptom:** the chat read-only lock (composer replaced by a "conversation is closed" banner) never engaged in normal use. The frontend logic was correct and the DTO exposed `IsClosed = Conversation.ClosedAt is not null`, but a full walk showed `isClosed` was always false — you could only see the banner by hand-editing `ClosedAt` in the DB.
**Cause:** `Conversation.ClosedAt` was **never assigned a non-null value anywhere in app code** — it existed as a column + a derivation (`IsClosed`, and `ChatTokens.StatusToken`'s closed override), but nothing ever WROTE it. ADR-001 said "closed once rental complete AND both reviews in," but no completion/review handler set it. Fix: `IConversationsStore.CloseForBookingAsync` (best-effort, idempotent) called from `ReviewsService` after each party-review submission (and `BookingsService.CompleteAsync` for ordering safety) once the booking is Completed and both party reviews are in.
**Rule:** this is the SAME class as M-007 (Completed booking → pill stuck on "active"): a **derived/read flag whose backing value has no writer**. When you find a computed read-only/status flag (`IsX = Field != null`, a derived status token), grep for where the backing field is ASSIGNED — if it's only ever set to its default/null, the feature is dead code and the "derivation is correct" is a false comfort. Every derived state needs a proven write-path exercised through a normal user flow (not a unit test with hand-set data, not a DB edit). Add an end-to-end test that drives the real trigger and asserts the flag flips.

## M-011: Port 4200 silently served a STALE docker prod build — live UI walk verified the wrong bundle
Date: 2026-07-11 | Area: infra / verification process (FIXED — container stopped; check is now part of walk setup)
**Symptom:** during the chat length-limit live walk, the freshly implemented composer counter never appeared and the walk "failed" — while unit tests and a code review said the feature was fine. `npm start` also failed with "Port 4200 is already in use". The app on :4200 responded 200 and looked completely normal.
**Cause:** the docker compose stack (`rental-api/docker-compose.yml`) was up (restart policy / Docker Desktop start), and its UI container publishes `4200:80` — nginx serving a **static prod build baked at image-build time** (days stale). Nothing distinguishes it visually from a dev server. Tell-tales: hashed bundle names in index.html (`main-XXXX.js` — `ng serve` serves unhashed `main.js`), `data-beasties-container` in the HTML (prerender pipeline), and `docker ps` showing `rental-api-ui-1 … 0.0.0.0:4200->80/tcp`.
**Rule:** before ANY live UI walk, prove the server on the target port serves the code under test: check `docker ps` for a UI container squatting the port, and/or grep the served index.html for hashed bundle names. If in doubt, start a fresh `ng serve` on a spare port (e.g. 4201) and walk that. A green-looking app is NOT evidence it's the right app — this is the dev-environment twin of M-008's meta-lesson ("verified live" is only as good as the environment).

## M-012: A state-transition event that fired under an OLD build is lost forever — derived state needs a self-heal path (3rd instance of the M-007/M-010 family)
Date: 2026-07-11 | Area: backend + process (FIXED — ChatService lazy close)
**Symptom:** in the docker stack a conversation stayed OPEN (both parties could message) although its booking was Completed and BOTH party reviews existed — exactly the state ADR-001 says must lock the chat. The M-010 write-path (`ReviewsService`/`BookingsService` → `CloseForBookingAsync`) was correct in the current code, all tests green.
**Cause:** the closing EVENT (the second review submission, 2026-07-10) happened while the docker API still ran an image built BEFORE the M-010 fix (merged 07-09, image rebuilt only 07-11). Event-driven writers only fire at the event; an event consumed by a build that lacked the writer never re-fires, so the derived state is permanently wrong for that row. Databases outlive deployments — dev DBs and prod DBs both accumulate rows whose lifecycle events predate the current code.
**Rule:** every event-driven derived state needs a SELF-HEAL path for rows whose event was missed: an opportunistic lazy re-check on read/write of the affected aggregate (here: `ChatService.SendMessageAsync` + `GetConversationAsync` re-derive "should be closed" and close idempotently — same pattern as the opportunistic `ExpirePendingAsync`). When you fix a missing write-path (M-010 class), ALWAYS ask "what about rows whose event already passed?" — a data backfill or a lazy re-check must ship with the writer fix. **Test-tooling footnote:** the docker walk first reported realtime "broken" because `locator.waitFor()` is strict-mode and the sent marker matched TWO elements (bubble + inbox snippet) — use `.first()` or `expect(locator).toBeVisible()` semantics when asserting on text that can legitimately appear in several places; a raw WS-frame probe (`page.on('websocket')`) is the ground truth for "did the push arrive".
