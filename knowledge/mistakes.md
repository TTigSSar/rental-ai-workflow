## M-002: A delegated agent seeded the EXEMPLAR feature, not the requested one
Date: 2026-07-07 | Area: infra (agent workflow)
**Symptom:** backend-dev was asked to add a **chat conversation** dev seed. It instead wrote a `SeedNotificationsAsync` method seeding NOTIFICATIONS (the feature it had read as a pattern exemplar), wired it into the seed runner, then died on a session limit before adding the requested get-or-create route. Orchestrator caught it on review and reverted the off-task change.
**Cause:** the spec pointed the agent at the notifications vertical as a "mirror this pattern" exemplar; the agent conflated "mirror the pattern" with "seed that feature's data." Cold subagents blur exemplar-vs-target when both are named in the same task.

## M-004: `takeUntilDestroyed()` bare in `ngOnInit()` → NG0203 runtime crash (green build/tests)
Date: 2026-07-07 | Area: frontend
**Symptom:** the chat thread page rendered blank on all viewports. `dotnet`/`npm` builds AND all unit tests were green — the failure only surfaced in the live walk (screenshots). Root cause: `takeUntilDestroyed()` called bare inside `ngOnInit` throws `NG0203: can only be used within an injection context`.
**Cause:** `takeUntilDestroyed()` needs an injection context (constructor / field initializer) unless a `DestroyRef` is passed explicitly. `ngOnInit` is not an injection context. Compiles fine; crashes at runtime. The chat feature had **zero unit tests**, so nothing caught it before the live walk.
**Rule:** call `takeUntilDestroyed()` in a field initializer/constructor, or pass `takeUntilDestroyed(this.destroyRef)` with an injected `DestroyRef`, when used in a lifecycle hook. Every new feature needs at least one component-render spec (`fixture.detectChanges()`) — it catches injection-context and template crashes that build+typecheck miss. This is why the two-tier verification's **live walk with screenshots** is mandatory: build+tests were all green here.

## M-005: Docker API crashes on restart — EF MigrateAsync race vs just-started SQL Server
Date: 2026-07-07 | Area: infra (docker)
**Symptom:** `docker compose up --build -d` after a `down` (volume kept) → `rental-api-api-1` exits **139**; API log shows `CREATE DATABASE [RentalPlatformDb]` → `SqlException (1801): Database 'RentalPlatformDb' already exists`, thrown inside `MigrateAsync` at `MigrationExtensions.ApplyMigrationsAsync` (unhandled → process dies). db + ui come up fine; api is missing from `docker compose ps`.
**Cause:** the db container's healthcheck is only `SELECT 1` on master, which passes before the persisted user DB is fully online. EF Core `Database.MigrateAsync()` runs its existence check in that window, gets "does not exist", tries `CREATE DATABASE`, and the DB *is* there → 1801 → unhandled crash. It's a startup race, NOT a code/schema bug — and **unrelated to whatever app change triggered the rebuild** (here it was a frontend-only nav icon).
**Rule:** operational fix — once db is fully online, restart just the API: `docker compose -f rental-api/docker-compose.yml up -d api` (succeeds on the retry). `down -v` (fresh volume) also avoids it but wipes data. **Proper fix (follow-up):** harden `MigrationExtensions.ApplyMigrationsAsync` with a retry loop / `EnableRetryOnFailure` so migrate retries transient SQL-startup errors instead of crashing the process. Don't chase it in app code — check the API log for 1801 first.

## M-006: Realtime echo + optimistic send → own message rendered twice (dedupe must be symmetric)
Date: 2026-07-08 | Area: frontend (realtime)
**Symptom:** with SignalR realtime chat, a user's OWN just-sent message rendered as two identical bubbles (persisted until reload); the server had only one record. All three live realtime scenarios otherwise PASSED — the two-session verifier walk caught this; the unit tests initially didn't (they tested the single, non-racing append path).
**Cause:** two append paths for the open thread — `sendMessageSuccess` (HTTP POST response) and `realtimeMessageResolved` (SignalR `messageReceived` echo, which the server broadcasts to BOTH participants incl. the sender). Only the realtime handler deduped by `id`; `sendMessageSuccess` appended unconditionally. When the hub broadcast beat the POST response (a real, fairly consistent race), the message was appended twice.
**Rule:** when a message can arrive via more than one channel (optimistic local echo, HTTP response, realtime push), **every** append path must dedupe by a stable server id — not just one. Broadcasting the sender's own message back over the socket is fine (simpler server), but the client must treat its own send and the echo as the same message. Add a regression test that seeds the message as already-present and asserts the second path is a no-op.

## M-007: Chat header status pill stuck on "Active" after completion — TWO root causes, and the orchestrator's first hypothesis was wrong
Date: 2026-07-08 | Area: backend + frontend (FIXED — rental-api `b2269d0`, Rental-Ui `5fca9cf`)
**Symptom:** the booking-context header pill atop the chat thread kept reading "Active" after the booking was Completed, and survived a full page reload. Surfaced by the verifier as a side-observation during the chat-system-messages walk.
**Cause (two, independent):** (1) **Backend, the real root cause** — `ChatTokens.StatusToken` mapped `BookingStatus.Completed => "active"`. The `closedAt`-override that would return `"closed"` never fires because the `Conversation.ClosedAt` auto-lock was never built. So the *server itself* derived `"active"` for a completed booking — the pill was stale even on reload. (2) **Frontend** — the pill binds `conversation.status`, set only by `loadConversationDetails`; realtime system messages appended to `messages` but never refreshed `status`, so even a corrected value wouldn't appear until reload.
**Rule:** the fix needed BOTH: backend `Completed => "closed"` (contract-safe — client already renders the token) for the value, and a frontend re-fetch on system-message arrival (`realtimeMessageReceived$` → `loadConversationDetails` when `isOpen && type==='system'`) for the live update. **Process lesson (the important one):** the orchestrator delegated the frontend fix on the hypothesis "re-fetch will re-derive the correct status" — that hypothesis was FALSE; the server returned the wrong value, so re-fetch alone rendered "Active" faithfully. The delegated agent's **live verification grounded against `GET /api/chat/conversations/{id}` (server truth)** falsified the orchestrator's root-cause assumption and found the real backend bug. Lesson: when a "derived" value looks stale, verify the derivation SOURCE (hit the API, read the token the server actually returns) before assuming the staleness is a client refresh problem. Don't trust the orchestrator's root-cause guess — make the verifier prove it against ground truth.

## M-008: SignalR realtime is dead in the docker stack — UI nginx doesn't proxy/upgrade the hub (405 on negotiate)
Date: 2026-07-08 | Area: infra (docker) — FIXED 2026-07-11 (Rental-Ui: `location /hubs/` + `nginx-map.conf` http-level `map` include; verified live in docker: negotiate 200, WS frames delivered to both parties)
**Symptom:** in the docker deployment the browser fails the SignalR hub negotiate with `Failed to complete negotiation … Status code '405' (Not Allowed)`; `messageReceived` never arrives, so realtime chat (new messages, system lines, the M-007 live pill flip) silently does nothing. Everything works under `npm start` (Angular dev proxy) — which is why every prior "realtime live" verification (realtime feature, system messages, M-007) passed: they all ran in dev, never against the docker UI container.
**Cause:** the UI container's nginx serves the SPA but has no `location` for the hub path (`/hubs/chat`) with the WebSocket upgrade headers (`Upgrade`/`Connection`, `proxy_http_version 1.1`) proxied to the API — so the hub endpoint returns 405 and the WS never upgrades.
**Rule:** OPEN backlog item — add an nginx `location /hubs/` block in the UI image proxying to the API service with WS upgrade headers (and confirm the API CORS/hub auth via `?access_token=` still works through it). Until then, realtime only works in dev. **Meta-lesson:** "verified live" is only as good as the environment — our live walks ran in dev, so a docker-only infra gap went unseen for three features. When a feature has a prod/docker deployment, verify at least once IN that deployment, not only in dev.

## M-009: Two-pane shell crashed on COLD direct-nav to /chat/:id — and skipping the independent verifier let it ship
Date: 2026-07-09 | Area: frontend + process
**Symptom:** direct-loading a thread URL `/chat/{conversationId}` in a fresh tab (no prior `/chat` visit) threw `Cannot read properties of undefined (reading 'paramMap')` and left the thread pane blank (footer showing). Warm nav (inbox → row click) worked, so it was invisible in normal use.
**Cause:** `ChatShellComponent.threadOpen` (the signal driving mobile single-column show/hide + the desktop empty pane) was seeded in a field-initializer by walking `this.route.firstChild` and reading `child.snapshot.paramMap`. On cold direct-nav a child `ActivatedRoute.snapshot` isn't populated yet during construction → throws. Fix: derive thread-open from the **router URL** instead (`isThreadOpenUrl(url)` = non-empty segment after `/chat/`), used identically in `startWith`/`initialValue`/`NavigationEnd` — no snapshot race, mirrors the app-level `isChatUrl`.
**Rule:** never read a child route's `snapshot` in a parent's field-initializer/constructor — it isn't ready on cold load; derive route-shape state from `router.url` (a string) instead. **Process lesson (the real one):** this shipped in the two-pane shell increment where the orchestrator MERGED ON THE IMPLEMENTER'S OWN LIVE WALK + a diff review, skipping an independent `verifier` pass to save a cycle — and the implementer only ever tested warm nav, so the cold-nav crash slipped. The independent verifier exists precisely to test the paths the implementer didn't think of (here: cold direct-nav). For any ROUTING/architectural change, run the independent verifier before merge even if the implementer self-verified — the classes of bug it catches (M-004, M-006, this) are exactly the ones the author is blind to. Also: every route-shape branch needs a cold-load test (seed the URL, assert no throw), which is why this had no regression test.

## M-010: A "derived read-flag" with no write-path — chat `IsClosed`/`ClosedAt` never fired (2nd instance of the M-007 pattern)
Date: 2026-07-09 | Area: backend (FIXED — rental-api `16237bc`)
**Symptom:** the chat read-only lock (composer replaced by a "conversation is closed" banner) never engaged in normal use. The frontend logic was correct and the DTO exposed `IsClosed = Conversation.ClosedAt is not null`, but a full walk showed `isClosed` was always false — you could only see the banner by hand-editing `ClosedAt` in the DB.
**Cause:** `Conversation.ClosedAt` was **never assigned a non-null value anywhere in app code** — it existed as a column + a derivation (`IsClosed`, and `ChatTokens.StatusToken`'s closed override), but nothing ever WROTE it. ADR-001 said "closed once rental complete AND both reviews in," but no completion/review handler set it. Fix: `IConversationsStore.CloseForBookingAsync` (best-effort, idempotent) called from `ReviewsService` after each party-review submission (and `BookingsService.CompleteAsync` for ordering safety) once the booking is Completed and both party reviews are in.
**Rule:** this is the SAME class as M-007 (Completed booking → pill stuck on "active"): a **derived/read flag whose backing value has no writer**. When you find a computed read-only/status flag (`IsX = Field != null`, a derived status token), grep for where the backing field is ASSIGNED — if it's only ever set to its default/null, the feature is dead code and the "derivation is correct" is a false comfort. Every derived state needs a proven write-path exercised through a normal user flow (not a unit test with hand-set data, not a DB edit). Add an end-to-end test that drives the real trigger and asserts the flag flips.

## M-011: Port 4200 silently served a STALE docker prod build — live UI walk verified the wrong bundle
Date: 2026-07-11 | Area: infra / verification process (FIXED — container stopped; check is now part of walk setup)
**Symptom:** during the chat length-limit live walk, the freshly implemented composer counter never appeared and the walk "failed" — while unit tests and a code review said the feature was fine. `npm start` also failed with "Port 4200 is already in use". The app on :4200 responded 200 and looked completely normal.
**Cause:** the docker compose stack (`rental-api/docker-compose.yml`) was up (restart policy / Docker Desktop start), and its UI container publishes `4200:80` — nginx serving a **static prod build baked at image-build time** (days stale). Nothing distinguishes it visually from a dev server. If in doubt, start a fresh `ng serve` on a spare port (e.g. 4201) and walk that. A green-looking app is NOT evidence it's the right app — this is the dev-environment twin of M-008's meta-lesson ("verified live" is only as good as the environment).

## M-012: A state-transition event that fired under an OLD build is lost forever — derived state needs a self-heal path (3rd instance of the M-007/M-010 family)
Date: 2026-07-11 | Area: backend + process (FIXED — ChatService lazy close)
**Symptom:** in the docker stack a conversation stayed OPEN (both parties could message) although its booking was Completed and BOTH party reviews existed — exactly the state ADR-001 says must lock the chat. The M-010 write-path (`ReviewsService`/`BookingsService` → `CloseForBookingAsync`) was correct in the current code, all tests green.
**Cause:** the closing EVENT (the second review submission, 2026-07-10) happened while the docker API still ran an image built BEFORE the M-010 fix (merged 07-09, image rebuilt only 07-11). Event-driven writers only fire at the event; an event consumed by a build that lacked the writer never re-fires, so the derived state is permanently wrong for that row. Databases outlive deployments — dev DBs and prod DBs both accumulate rows whose lifecycle events predate the current code.
**Rule:** every event-driven derived state needs a SELF-HEAL path for rows whose event was missed: an opportunistic lazy re-check on read/write of the affected aggregate (here: `ChatService.SendMessageAsync` + `GetConversationAsync` re-derive "should be closed" and close idempotently — same pattern as the opportunistic `ExpirePendingAsync`). When you fix a missing write-path (M-010 class), ALWAYS ask "what about rows whose event already passed?" — a data backfill or a lazy re-check must ship with the writer fix. **Test-tooling footnote:** the docker walk first reported realtime "broken" because `locator.waitFor()` is strict-mode and the sent marker matched TWO elements (bubble + inbox snippet) — use `.first()` or `expect(locator).toBeVisible()` semantics when asserting on text that can legitimately appear in several places; a raw WS-frame probe (`page.on('websocket')`) is the ground truth for "did the push arrive".

## M-013: The layer every test fakes is the layer nobody tests — chat image upload was 100% broken with 453/453 tests green
Date: 2026-07-11 | Area: backend + infra (FIXED, pre-merge — caught by the independent verifier)
**Symptom:** the chat image-attachment vertical shipped from the implementers with `dotnet test` 195/195 and `npm test` 258/258 green, builds clean. It was **completely non-functional in both environments** — every single upload failed. The independent verifier's live walk found it in minutes.
**Cause (two, in the same layer):**
(1) **Windows** — `LocalFileStorageService` built its target-directory field with a bare `Path.Combine`, so it kept the literal forward slash from the config value `"uploads/chat"` (`...\wwwroot\uploads/chat`), while the candidate path went through `Path.GetFullPath` (all `\`). The `StartsWith` containment check compared half-normalized against fully-normalized → **always false** → `InvalidOperationException` → HTTP 500.
(2) **Docker** — `docker-compose.yml` gained a `chat-uploads` volume, but the `Dockerfile` still only `mkdir`+`chown`'d `uploads/listings`. Docker auto-created the missing mount point as `root:root`; the container runs as `appuser` → `Permission denied` on every write.
**Why no test caught either:** every existing test injects `FakeFileStorageService`. The real `LocalFileStorageService` had **zero tests**, so the one class where both bugs lived was the one class the suite never executed. A fake at the boundary makes the boundary invisible.
**Rule:** **the component you fake in every test is the component you must test for real, at least once.** Any adapter to the outside world (filesystem, clock, storage, network) needs at least one test against the REAL implementation, or its bugs are structurally unreachable — green suites will actively reassure you. Corollaries proven here:
- **A new volume in `docker-compose.yml` is only half a mount.** The `Dockerfile` must create and `chown` the directory *before* the volume mounts over it, or the mount point is root-owned and the app can't write. A wrong mount looks fixed but isn't (same trap as M-008).
- **Copying an existing pattern copies its bugs.** The Windows path bug was pre-existing in `SaveListingImageAsync` — listing image upload was *also* 500ing on Windows dev and nobody had noticed. The new code faithfully reproduced it. When mirroring an exemplar, the exemplar is not evidence of correctness.
- The fix also closed a latent traversal hole nobody had flagged: a bare `StartsWith(root)` accepts a **sibling** directory (`uploads/listings-evil` reads as "inside" `uploads/listings`). Containment must compare on a path boundary (`root` itself, or `root + DirectorySeparatorChar`).
**Process note (the point of the pipeline):** the orchestrator did NOT merge on the implementers' self-reported green — the independent `verifier` ran the live walk in **both** dev and docker (per M-008) and returned FAIL with reproductions. This is M-009's lesson applied correctly instead of learned again.

## M-014: Compose healthcheck probed `localhost` — busybox resolved it to `::1`, nginx listens IPv4-only, container "unhealthy" forever
Date: 2026-07-13 | Area: infra / deploy (FIXED — rental-api `76cd6ed`, found live on the production VPS)
**Symptom:** first production deploy on the VPS: `db` and `api` healthy, the app fully working (curl from the host returned 200), but `ui` sat permanently `unhealthy` — and because `cloudflared` gates on `depends_on: condition: service_healthy`, the tunnel container refused to start at all: `dependency failed to start: container rental-api-ui-1 is unhealthy`. No error in any service log points at the cause; only `docker inspect` of the health log shows `wget: can't connect to remote host: Connection refused`.
**Cause:** the `ui` healthcheck was `wget --spider http://localhost:80/`. Busybox wget (nginx:alpine) resolves `localhost` to IPv6 `::1` first and does not fall back; the container's nginx config has only `listen 80;` (IPv4, no `listen [::]:80`). So the probe connected to `::1:80` where nothing listens — refused forever, while the service itself was fine on IPv4.
**Rule:** compose/docker healthchecks must target an explicit IPv4 literal (`http://127.0.0.1:PORT/`), never `localhost`, unless the service provably listens on `::`. Two generalizations: (1) a healthcheck is code — validate a new probe against a real running container before trusting it (the `api` probe was pre-validated live and was correct: bash `/dev/tcp/127.0.0.1/8080`); (2) when a service gates on `service_healthy`, a broken *probe* is indistinguishable from a broken *service* from the dependent's point of view — when "X failed to start because Y is unhealthy", inspect Y's `State.Health.Log` before touching Y's actual workload.

## M-015: A check that grades the LAST LINE of a file written by two streams grades the wrong thing — smoke.sh WARNed on every SUCCESSFUL backup
Date: 2026-07-14 | Area: infra / monitoring (FIXED — rental-api `33618a5`)
**Symptom:** `deploy/smoke.sh`'s `backup-log` check emitted a WARN on every run, on a server where the nightly backup was in fact succeeding (the `.bak` was fresh and the separate `backup-freshness` check passed).
**First (wrong) diagnosis — by the agent AND the orchestrator:** "the two scripts disagree on the log-line format." They don't. `backup-production.sh` already writes a canonical machine-readable entry (`<ts> OK db=… uploads=…`), and smoke.sh's pattern matched it fine.
**Actual cause:** `backup.log` has **two interleaved writers**. The crontab entry redirects the script's console output into the very file the script writes structured entries to (`backup-production.sh >> /opt/dorent/backups/backup.log 2>&1`). A successful run therefore ends with human prose (`Backup OK: … (612K)`) printed *after* the canonical entry — plus sqlcmd chatter. `tail -n 1` graded the prose, not the outcome.
**Fix:** smoke.sh now selects the **last line matching the canonical form** (anchored timestamp + status token) and derives PASS/WARN from the captured status group, ignoring non-canonical prose (including historic lines). The canonical format is documented as an explicit contract above `log()` in `backup-production.sh`: change it there → update `canonical_re` in the same commit.
**Rule:** never grade a monitoring signal by the last *physical* line of a file that more than one stream writes to — grade the last line that matches your **structured** contract. Two corollaries, both learned the hard way here: (1) **a warning that always fires is worse than no check** — it trains everyone to ignore the warning channel, so a permanently-yellow check is a bug to fix now, not later; (2) the first plausible explanation of a check misfiring ("format mismatch") is not the explanation — the agent only found the real cause because it re-read both scripts and the live log instead of trusting the earlier report. Underlying design smell (still open, deferred to a card): merging stdout+stderr into a structured logfile at all.

## M-016: `GetValue<bool>` + an empty env var = crashloop; compose sets unset variables to "", it does not omit them (caught pre-deploy)
Date: 2026-07-16 | Area: backend + infra (FIXED pre-deploy — compose default `e22864e`, tolerant parse in the follow-up)
**Symptom (never reached production — found by platform-engineer while wiring the new `Bootstrap__DemoContent*` vars):** deploying the demo-content feature with the **documented default** config would have crashlooped the `api` container on startup, taking the WHOLE SITE down (`ui` and `cloudflared` gate on api's `service_healthy`, so a dead api = no site, not a missing feature).
**Cause, in two halves — neither obvious alone:** (1) Docker Compose does **not** omit an unset/blank variable: `${VAR}` with an empty `.env` key renders `Bootstrap__DemoContentEnabled: ""` — the key is *set, to an empty string*, which is a different thing from absent. (2) The app read it with `configuration.GetValue<bool>(...)` — a non-nullable value type. The real binder's behaviour (probed, not assumed): absent → `false`; **empty string → throws** `InvalidOperationException`; `yes`/`1`/`on`/any typo → throws. And `.env.production.example` shipped the key *empty*. So the happy path was an outage.
**Fix:** `${BOOTSTRAP_DEMO_CONTENT_ENABLED:-false}` in compose (covers unset AND empty) + a tolerant `bool.TryParse` in the app that logs a loud WARNING and degrades to "feature off" instead of throwing (compose cannot defend against a typo'd non-empty value; only the app can).
**Rule:** an **optional** feature flag must never be able to kill the process — parse it tolerantly, log loudly, degrade to off. Corollaries: (a) never read env-var-backed config into a **non-nullable value type** without a fallback — the empty string is the default state of a compose-mapped variable, not an exotic edge case; (b) in a stack where dependents gate on `service_healthy`, ANY startup throw is a full outage, so the startup path deserves paranoid config handling; (c) this was found only because the agent was told to *check* the unset/empty behaviour instead of mirroring the existing pattern — the existing admin-bootstrap vars are plain strings and looked like a safe precedent, which is exactly M-013's "copying an existing pattern copies its assumptions".

## M-017: A Trello card's own description asserted a privacy gate that never existed — and the whole feature was built on top of that assertion
Date: 2026-07-21 | Area: backend + contract (FIXED in `875790f`, same day)
**Symptom:** Maps Phase 1 shipped a deliberately-fuzzed public coordinate (geohash-6 centroid, ~933 × 611 m at Yerevan's latitude), a detail-page map drawing a 600 m uncertainty circle, and a create-listing form promising the owner in EN/HY/RU that "only the approximate area is published, never your exact address" — while `ListingDetailsResponse.AddressLine` was returned **ungated to anonymous callers** in the same JSON response. The exact street address sat next to the fuzzed coordinates for the entire phase.
**Cause:** the roadmap card H1 opened with "Public `ListingDetailsResponse` returns raw Latitude/Longitude **while AddressLine is gated behind contactRevealed**". That clause was simply false for this endpoint — a `contactRevealed` gate did exist, but only in `BookingsService.MapBookingDetail`, a different code path. The card was written by an agent doing design analysis, read as a statement of fact by every agent that came after, and never verified: H1's own implementer audited every DTO for *coordinates* (the thing it was asked about) and correctly reported "no other leak", because `AddressLine` was framed as already-solved background, not as something to check. It was found four cards later, incidentally, by an agent reading adjacent code for an unrelated task.
**Rule:** a claim about what the code currently does is **not** evidence, no matter how authoritative the document sounds or who wrote it — this is the M-001 family (a document describing intent read as describing reality), but the new twist is that the false claim was inside the *task specification itself*, in the sentence establishing why the task mattered. Two corollaries: (a) when a spec's premise says "X is already safe", verify X before building on it — the premise is the least-reviewed sentence in any card and the most load-bearing; (b) when a feature's entire value is a privacy boundary, audit the **whole response payload** for that class of data, not just the field named in the ticket — the gate is only as good as the least-gated field sharing the response.

## M-018: Playwright's `reuseExistingServer` will happily reuse a stale Docker container on port 4200 — the suite goes green without testing your code
Date: 2026-07-22 | Area: frontend + testing (OPEN — no guard added yet)
**Symptom:** during the tile-provider change, `npm run e2e` reported 8/8 green. The run was worthless: port 4200 was held not by a fresh `ng serve` but by the **`rental-api-ui-1` Docker container**, an nginx serving a production build baked ~30 minutes earlier. Playwright's `webServer.reuseExistingServer` saw *something* answering on 4200, decided its job was done, and ran the whole suite against a build that predated every line under test.
**Cause:** `reuseExistingServer` probes for a listening port, not for *the right server*. Any process on 4200 satisfies it — a stale container, a colleague's dev server, a forgotten instance from another branch. The mocked e2e tier has no equivalent of the guard the `real` tier already carries. This is the M-011 family: the check verified something adjacent to what it claimed to verify.
**Rule:** a green suite is only evidence if you can name what it ran against. When an e2e run is the thing standing between a change and "verified", confirm what is actually serving the port (`docker ps`, `netstat -ano` + the owning PID) before trusting the result — especially on this machine, where a full docker stack and `ng serve` compete for 4200. The agent that hit this caught it only because it went looking; the default outcome was a false green. Worth a `webServer` guard that asserts the server's identity (a build-stamp header or a dev-only endpoint) rather than merely its existence.

## M-019: Angular's default `ViewEncapsulation.Emulated` silently voids third-party CSS for any library that builds its own DOM
Date: 2026-07-22 | Area: frontend (FIXED in `d988e39`)
**Symptom:** the Leaflet map rendered real map imagery, but only in patches — correctly-aligned 256 px tiles with large grey gaps between them, on both the picker and the listing-detail map. Two earlier explanations were plausible and both wrong: the OSM "418 Access blocked" throttling (tiles probed fine, 200 + real PNG bodies), and the dialog-animation race against `app-map`'s two fixed `setTimeout(invalidateSize)` calls.
**Cause:** `MapComponent` used Angular's default `ViewEncapsulation.Emulated`, which rewrites every selector in the component's compiled stylesheet to require an `_ngcontent-*` attribute — **including third-party CSS pulled in with `@use 'leaflet/dist/leaflet.css'`**. Angular stamps that attribute onto nodes it creates from the template. Leaflet creates its tiles, panes and controls imperatively with `document.createElement`, so they never receive it. `.leaflet-tile { position: absolute }` therefore compiled into a selector that could not match a single real tile, and every tile fell back to the `<img>` UA default (`position: static; display: inline`). Tiles stacked in normal document flow and the container's `overflow: hidden` clipped the column — producing a pattern that looks exactly like "some tiles failed to load".
**Rule:** when a component wraps a library that builds DOM outside Angular's templates — Leaflet, chart libraries, editors, anything imperative — its stylesheet cannot be emulated-encapsulated. Use `ViewEncapsulation.None` (and accept that the component's own classes become global, so namespace them) or `::ng-deep`. Two corollaries: (a) **"CSS is loaded" and "CSS applies" are different claims** — the stylesheet was present and parsed the whole time, and a network check for it proves nothing; verify the computed style of an actual node instead; (b) the agent found this only by falsifying the leading hypothesis outright — it disabled both `invalidateSize` timers and reproduced the bug byte-for-byte, which is what turned attention away from timing. A hypothesis handed down in a task description deserves a disproof attempt, not a search for confirmation (cf. M-017).

## M-020: Three contract-drift bugs reached production because every layer that could have caught them was mocked against itself
Date: 2026-07-23 | Area: contract + frontend + testing (two FIXED in `2169ef6`/`7aabd02`, one carded)
**Symptom:** production nginx logs on dorent.am showed `GET /api/profile/me` returning **404 to two different real logged-in users**, ten times in 36 hours. The endpoint had never existed: `api-contract.ts` declared `profile.me`, `ProfileApiService` called it, and `rental-api` had no `ProfileController` at all. A full contract audit triggered by that finding turned up two more of the same class — the renter review summary was typed as `OwnerReviewSummary`, so `all-reviews-page` read `pickupHandoverAverage`/`friendlinessAverage` off a `RenterReviewSummaryResponse` that actually carries `returnedOnTimeAverage`/`careOfToyAverage`/`wouldRentAgainAverage` (two rating bars silently rendered from `undefined`, a third never rendered), and the listings search box sends `?title=` to a `ListingsQueryFilter` with no such property, so **the search returns unfiltered results**.
**Cause:** not one bug but one missing check, three times. Nothing in the stack ever compares the declared contract against the real routes. Worse, three independent mechanisms each converted a hard failure into a silent one: (a) `profile.effects.ts` has a `catchError` fallback that quietly substitutes auth-store data, so the 404 page still rendered — just without phone and language; (b) TypeScript cannot catch a wrong DTO type when the wrong type is a *structurally valid* interface, so `undefined` flowed into the template as an empty bar; (c) ASP.NET model binding **discards unrecognised query keys without error**, so an unsupported filter degrades into "no filter" rather than a 400. The unit spec that pinned the profile URL was no defence at all: `HttpTestingController` answers whatever URL the same test declares, so it asserts "the UI requested what we wrote down", never "a server answers there" — a client-only path and a real path are indistinguishable to that harness by construction (the M-013 gap, reproduced).
**Rule:** a contract is only real if something executes both sides against each other. Pin it with a check that reads the **server's** route table — `e2e/real/api-contract-conformance.spec.ts` now walks every path in `ApiContract` against the live OpenAPI document, so this whole class fails in CI instead of in nginx logs. Three corollaries: (a) **a defensive fallback around a network call is an outage-hiding device** — `catchError` that substitutes plausible data means the failure never surfaces anywhere except the server's access log, so pair any such fallback with telemetry that fires when it engages; (b) when two endpoints return "the same kind of thing" (owner reviews, renter reviews), reusing one's TypeScript interface for the other is a coin flip that TypeScript cannot grade — check the actual DTOs, and note that the wrong code here carried an explicit comment *asserting* the shapes were shared, which is M-017's "a claim about what the code does is not evidence"; (c) the API container logs **zero HTTP requests** (`HttpLogging` off, only EF Core SQL), which is why a user-facing 404 was discoverable solely from nginx — the observability gap is what made the detection accidental rather than routine.

## M-021: A test that asserts on a widget's CSS class instead of the outcome it controls races Angular's change detection — and a display-format ADR quietly rotted a second assertion in the same file
Date: 2026-07-23 | Area: frontend + testing (FIXED in `b32b266`, Rental-Ui)
**Symptom:** `e2e/real/booking-lifecycle.spec.ts` — the only real-stack booking journey, the suite that is supposed to gate merges — had been red for some time. Two unrelated defects lived in the one file and **masked each other**: run the spec alone and the calendar helper failed first; run the full `--project=real` and the price assertion failed first. Each investigation that looked at only one run mode saw only one bug and, seeing a plausible cause, stopped.
**Cause (defect 1 — the interesting one):** `selectRange()` cleared the date range by clicking a quick-book chip, then read `clearChip.getAttribute('class')` **immediately after `.click()`** to decide whether a second click was needed. Angular repaints a `[class.x--active]` binding on the change-detection pass *after* the click event, not synchronously with it, so that read returned the **stale pre-click class** at effectively random. A wrong answer left the chip toggled the wrong way, and every subsequent `toPass()` retry then re-entered from the poisoned state — the box never cleared, and the 90 s `toPass` budget expired. The app was innocent: a diagnostic script driving the same toggle with a race-free check ran 20/20 clean. Note `getAttribute` is one of the few Playwright calls with **no auto-retry** — it is a raw one-shot DOM read wearing the same API shape as the auto-retrying `expect`s around it, which is exactly why it slid in unnoticed.
**Cause (defect 2):** the price assertion still matched `/\$36(\.00)?/` long after ADR-007 moved the app to dram-only pricing; the app renders `"14,000 ֏"`. The fixture's `TOY_KITCHEN.pricePerDay` had likewise been left at `9` with a "$9/day" comment while the seed says 3,500 AMD. Nothing failed loudly at the time because the file was already red for the other reason.
**Rule:** assert on **the state the user sees**, never on the internal marker that is supposed to produce it — the class attribute is an implementation detail one change-detection tick behind the truth, while the pickup box's own text is the thing the test actually cares about and comes with auto-retry for free. Three corollaries: (a) inside a `toPass()`/retry block, any step that can leave the page in a *worse* state than it found it turns the retry loop from a remedy into an amplifier — make each iteration idempotent from any starting state, not just from the happy one; (b) when one file holds two failures, fix both and then re-verify in **both** run modes — order-dependent masking is why this survived multiple looks; (c) a cross-cutting display decision (currency, date format, units) rots every hardcoded string in the test suite, so derive the expectation from the same rule the app uses (here: the seed price × days, formatted the way `DramCurrencyPipe` formats) instead of pinning a literal that no one will think to grep when the ADR lands. And, as in M-018: a red suite that everyone has learned to step around is not a lesser problem than no suite — it is a worse one.

## M-021: A navigation that replaces the whole query string silently wipes every filter its own component does not know about
Date: 2026-07-24 | Area: frontend (FIXED in `2873ef4`)
**Symptom:** on `/listings` at mobile width, opening the filter sheet and tapping **Apply with no changes at all** erased `ageGroup` and `maxDistance` from the URL. `/listings?ageGroup=0-12&maxDistance=5` became a bare `/listings`, and the result set silently widened. Found by `qa-engineer` while assessing regression coverage for the district filter — reproduced live against the Docker stack, not inferred from reading code.
**Cause:** `ListingsFiltersComponent` navigated with `queryParamsHandling: 'replace'`, and its `toQueryParams()` returned a hardcoded object of exactly the six keys that component owns. `ageGroup` and `maxDistance` are rendered only by the **desktop sidebar** in `ListingsPageComponent`, so they were absent from that object and `replace` dropped them. The trigger was `applySheet()` calling `patchValue()` without `{ emitEvent: false }`, which fired `valueChanges` even when nothing changed — so a no-op tap was enough. Every other navigation in the same feature already used `'merge'`; the sheet was the lone outlier, which is why nobody noticed the convention had an exception.
**Rule:** once a feature has **more than one UI surface writing the same URL state**, `queryParamsHandling: 'replace'` is a bug waiting to happen — it makes each surface responsible for enumerating params it has no reason to know exist, and that list silently rots as the other surface grows. Default to `'merge'` and express clearing as an explicit `null` (Angular's Router deletes a key on `null`, leaves an absent key alone), which handles set and unset without the component needing a complete inventory. Two corollaries: (a) a filter split across a mobile sheet and a desktop sidebar is exactly the shape that breeds this — the bug pre-dated the district filter, but adding a third surface (P1-7) is what made anyone look; (b) `patchValue`/`setValue` fire `valueChanges` by default, so "apply" handlers can perform a full navigation for a no-op interaction — pass `{ emitEvent: false }` unless the emission is the point.
